# ギャップ分析: リスク分析報告書 vs 現行検証環境

## 0. この文書の目的

外部リスク分析報告書（Flashback SCN＋Data Pump＋Archived Redo差分適用方式の
リスク分析とベストプラクティス）の要件に対し、現行の検証環境実装が
どこまで達成できているか、何が不足しているかを棚卸しする。

実装方針の決定はこの文書では行わない（ギャップの可視化のみ）。
関連: `docs/migration-strategy.md` / `docs/delta-extract-design.md`

---

## 1. 報告書の核心: 三分割アーキテクチャ

報告書の中核的主張は「初期ロード・差分反映・スキーマ変換を1経路で済ませると
必ず破綻する。三つに分割せよ」である。

```
① 基準SCN整合性   → Data Pump（FLASHBACK_SCN で整合点固定）
② コミット順差分   → LogMiner（COMMIT_SCN 基準・バッチ適用）
③ スキーマ差分吸収 → RAW/STG スキーマ + 決定論的変換レイヤ
```

> 「この三つを一つの経路で済ませようとすると、ほぼ確実に
>  『境界トランザクション』『DDL混在』『変換対象テーブル』の
>  どこかで破綻する」（報告書 結論）

現行の `delta-extract` 方式は **③ RAW/STG 経由という方向性は報告書と一致**
している。報告書はむしろこの設計判断を裏付けている。
ただし **② の境界条件（COMMIT_SCN）の作り込みが不足**している。

## 3. ギャップの優先度評価

報告書は「方式採用そのものより、境界条件と運用設計の実装が重要」と繰り返す。
ギャップを「破綻の爆発半径」で並べると以下になる。

### 最優先（ここが誤ると差分全体が信頼できなくなる）

- **G2: COMMIT_SCN 境界** — 長時間Txの欠落/重複。データ不整合の温床
- **G3: COMMITTED_DATA_ONLY** — ロールバックされた変更の誤適用
- **G4: commit_scn 台帳** — 障害再開時の欠落/二重適用

> この3つは「②コミット順差分」の正しさそのもの。
> 現行の単純SCNフィルタは検証用としては動くが、本番境界条件を満たさない。

### 高（三分割の完成に必須）

- **G5: STAGING→TARGET 変換層** — ③の後半が空（タスク#5）
- **G6: テーブル3分類** — 経路設計の前提
- **G7: DDL凍結** — 辞書不一致の検知・防止
- **G12: 二段階検証** — サイレント不整合の検出

### 中（運用条件の確立）

- **G1: FLASHBACK_SCN 初期ロード**（タスク#5と連動）
- **G9/G10: UNDO・Archive 運用**（タスク#4で計測中）
- **G13: LOBフォールバック**
- **G14: 再開Runbook**

### 本番固有（検証環境ではスコープ外だが本番必須）

- **G15: セキュリティ**（暗号化・権限・ライセンス）

## 4. 現行実装が報告書と「合致している」点（評価できる点）

ギャップだけでなく、既に正しい方向にある点も記録する。

| 項目 | 評価 |
|------|------|
| RAW/STG 経由設計（③の骨格） | 報告書の中核推奨と一致。delta_queue→STAGINGの分離は正しい |
| 移行元でLogMiner解析・結果SQLを搬送 | 辞書をソース側に置く正しい設計（flat-file辞書のPDB問題で実証済み） |
| continuous_mine 不使用のバッチ設計 | 19c前提と一致 |
| Supplemental Logging ALL COLUMNS | 差分再構成の前提を満たす |
| Data Pump ファイル搬送（オフライン対応） | AWS不可・DBリンク困難の制約に合致 |
| applied_at による冪等適用 | 部分的だが冪等性の土台はある |

---

## 5. 結論

現行検証環境は **三分割アーキテクチャの「①の一部」と「③の前半」を実装済み**で、
方向性は報告書と整合している。

決定的に不足しているのは **「②コミット順差分の境界条件」**（G2/G3/G4）であり、
これは本番適用の可否を左右する最重要ギャップである。
現行の `SCN > last_scn` 単純フィルタは検証貫通には十分だが、
本番の長時間トランザクション・障害再開には耐えない。

次点で **「③の後半＝STAGING→TARGET変換層」**（G5/G6）が三分割を完成させる鍵。

運用条件（UNDO/Archive/DDL凍結/検証）は、PoC を通じて
報告書が示す監視指標・テストケースで実証していく領域である。
